# Category Entry Memo 開発仕様書（仕様および取り決め）

本ドキュメントでは、個人用Markdownメモを効率的に管理・閲覧・編集するためのデスクトップ向けWebアプリケーション **「entry-memo」** を開発・保守する上で必要な仕様と設計上の取り決めについて規定します。

---

## 1. アプリケーションの目的と基本モデル

個人用Markdownメモを、カテゴリーとエントリー（ファイル）の階層構造で整理し、常に「概要」（最新の結論・方針）を最上部に表示しながら、その配下に「ブロック」（時系列ログや各種考察）をカード状に蓄積・可動させるための、キーボード主体の軽量なメモ管理環境を提供する。

### 基本的なデータモデル
```text
カテゴリー（ディレクトリ）
  └ エントリー（Markdownファイル）
      ├ 概要（現在の結論・方針・未解決事項などを記述する必須セクション）
      └ ブロック（H3見出し単位で分割された、並び替え・移動可能なカードブロック）
```

---

## 2. 開発・実装の最重要方針

* **Web標準技術のみ**: 動作環境は HTML, CSS, JavaScript (Vanilla JS) のみとする。
* **外部非依存**: 外部ライブラリ、外部CDN、外部Webフォント、外部APIを一切使用せず、ネットワーク通信を行わないスタンドアロンアプリとして動作する。
* **ロータリーファイルアクセス**: `file://` スキームで `index.html` をブラウザで直接起動し、**File System Access API** (`showDirectoryPicker`) を使ってローカルの特定フォルダを読み書きする。
* **グローバル名前空間の管理**: グローバルな汚染を避けるため、アプリのインスタンスや設定はすべて単一のオブジェクト `window.EntryMemo` の配下にカプセル化する。
* **セキュリティ（HTMLエスケープ）**: MarkdownをそのままHTMLとしてブラウザに評価・描画させず、スクリプト実行やタグ埋め込みの危険を防ぐため、必ずHTMLエスケープされたテキストとして表示する。
* **自動修復の禁止**: フォーマット規約に従っていない不正なMarkdownファイルを検出した場合は「編集不可」として警告し、アプリ側で勝手に中身を整形・上書きしてはならない。

### ファイル構成と依存関係
```text
entry-memo/
├─ index.html
├─ style.css
├─ app.js
├─ README.md
├─ fixtures/
│  └─ demo-data.js
└─ src/
   ├─ utils.js
   ├─ markdown.js
   ├─ storage.js
   └─ ui.js
```
HTMLでのscriptタグ読み込みは、以下の依存順で行うこと。
1. `fixtures/demo-data.js` (デモデータ)
2. `src/utils.js` (文字列処理、ID生成ヘルパー)
3. `src/markdown.js` (Markdownパーサー・シリアライザー)
4. `src/storage.js` (ストレージアダプタ。FileSystem / Demo)
5. `src/ui.js` (UI制御、イベントリスナー、キーボード移動)
6. `app.js` (アプリケーションエントリポイント、ビジネスロジック)

## 3. Markdown構造・規約

各エントリーは、必ず以下の厳格なMarkdown構造を満たす必要がある。

```markdown
# エントリータイトル

## 概要

- ここに概要情報を記述（空でも可）

## ブロック

### [a73bc] 2026-06-23 09:00 ブロックタイトル

ここに本文を記述する。
H4以下の見出し（#### 小見出し）や引用、箇条書きを含められる。
```

### 仕様ルール
1. **見出しレベルルール**: H1は「エントリータイトル」、`## 概要` と `## ブロック` は必須のセクション見出しとする。
2. **ブロック (H3)**: 1件のブロックは、`### [ID] [タイムスタンプ（任意）] ブロックタイトル` の見出しから、次のH3見出しの直前またはファイル末尾までとする。
3. **ブロックID**: `[a73bc]` のように英小文字と数字からなる一意な5文字の不変IDとする。編集や他のエントリーへの移動を行っても、このIDは変更しない。
4. **ブロック本文内でのH3禁止**: ブロックの本文内で見出しを使う場合は、必ずH4以下（`####`）とする。
5. **タイムスタンプ**: 新規にブロックを追加した際、自動的に `yyyy-mm-dd hh:mm` の形式でIDの直後にタイムスタンプを付与する。

## 4. UIおよびキーボード操作の取り決め

キーボードのみでのマウスフリーな操作を実現するため、移動およびフォーカスの振る舞いに以下の仕様を適用する。

### 4-1. J/Kキーによるフォーカス移動とループ挙動
* **エントリー詳細画面内**:
  * ナビゲーションのインデックスとして、`0` に「概要セクション」、`1 〜 blocks.length` に各「ブロックカード」を割り当てる。
  * 概要セクションがフォーカスされた際、視覚的なフィードバックを与えるため `.summary-box` の境界線（青）とシャドウを強調する（`.summary-section.focused` スタイル）。
  * **ループ仕様**: 最後のブロックカードで `J`（下）を押したときは「概要」へ移動し、概要で `K`（上）を押したときは「最後のブロックカード」へ移動する。
  * ブロックカードが無い場合は、概要にのみフォーカスが留まる。
* **エントリー一覧画面内**:
  * `J`/`K` キーでカテゴリーの中のエントリーカードを上下に選択移動する。
  * **ループ仕様**: 一番下のカードで `J` を押したときは一番上に戻り、一番上のカードで `K` を押したときは一番下のカードに移動する。
* **初期移動**:
  * 選択インデックスが `-1`（未フォーカス）の初期状態から `J` を押した際は一番上（エントリー詳細なら概要、エントリー一覧なら最初のエントリーカード）を選択し、`K` を押した際は一番下（エントリー詳細なら最後のブロックカード、エントリー一覧なら最後のエントリーカード）を選択する。

### 4-2. エントリーの編集・削除
* **エントリーの削除 (ゴミ箱移動)**:
  * エントリーが `ゴミ箱`（または `trash`）以外のカテゴリーにある時に削除を実行した場合、確認ダイアログを経て `ゴミ箱` カテゴリー（ディレクトリ）へエントリーファイルを移動（リロケート）させる。
  * エントリーがすでに `ゴミ箱` カテゴリーにある時に削除を実行した場合は、確認ダイアログを経て、ファイルシステムから永続的にファイルを削除する。
* **エントリーの編集**:
  * エントリー編集モーダルを介して、エントリーの「タイトル」および「所属するカテゴリー」を変更できる。
  * タイトルが変更された場合、Markdown内の H1 ヘッダーを自動更新し、ファイル名も新しいタイトルに合わせて自動リネームする（重複時は連番を解決）。
  * カテゴリー名が変更された場合、ファイルを物理的に別のカテゴリーフォルダへ移動（リロケート）する。新しく「新しいカテゴリー」をその場で作って移動させることも可能。

### 4-3. ブロックのマージ機能
* 各ブロックカードにあるチェックボックス（`X` キーでトグル）で複数選択し、1つのブロックにマージして新規エントリーを切り出せる。
* マージする際、元の各ブロックの見出し（H3: `###`）は1段デモートされて H4（`####`）になり、元の本文内の見出しレベルも連動してレベルが1段下げられる。
* 新規エントリーの保存成功を確認した後にのみ、マージ元のエントリーから該当するブロック群を安全に削除する。

### 4-4. オートリサイズ入力
* 概要編集のテキストエリアおよびブロック本文編集のテキストエリアは、入力されたテキスト量（行数）に合わせて高さが自動的に増減（オートリサイズ）する。
* テキストエリアの最小高（`min-height`）はCSS側で確保し、手動リサイズ（`resize`）およびスクロールバー（`overflow-y`）は無効化する。

## 6. Webサーバー環境への移植要件

本アプリをWebサーバー環境へホストする場合の設計方針は [specification_for_api.md](file:///app/specification_for_api.md) に準拠する。

* **バックエンドAPIの構築・設置**: サーバー上のファイルを操作する軽量なバックエンドAPIを設置。
* **`ServerStorage` アダプタの適用**: `src/storage.js` に `ServerStorage` を追加し、HTTP `fetch` にてAPIサーバーを経由して読み書きする。
* **初期起動フローの分岐**: アプリ起動URLが `http(s)://` の場合、`ServerStorage` を自動セットして `showDirectoryPicker` をスルーすることで、ユーザーはフォルダ選択の手間なくサーバーホスト環境を利用できる。
