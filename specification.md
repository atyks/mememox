# mememox 設計・動作仕様書 (System Specification)

本書は、Markdownファイルをデータストアとして使用する、カード型のブロック構造を持つ軽量・安全なエントリー型メモシステム `mememox` の全体設計および実装仕様を定義した文書です。

> [!NOTE]
> **キーボードショートカットのカスタマイズについて**
> 本書およびマニュアルに記載されているキーボードショートカットは、いずれも**デフォルトの登録値**です。
> アプリケーションは `app/src/keymaps.js` を動的にロードしてキー割り当てを任意に上書きする機能を備えています。
> カスタマイズが行われている場合、アプリ内で `?` キーを押して表示されるヘルプ画面には、設定されたカスタムキーが**自動反映・同期描画**されます。

---

## 1. システム概要・基本コンセプト

`mememox` は、情報の陳腐化を防ぎ、常に「現在の結論や方針（概要）」と「時系列の作業・思考ログ（ブロック）」を一体化して管理するためのパーソナルナレッジマネジメント（PKM）ツールです。

### 主な特徴
1. **Markdownベース**: すべてのメモは、規定のMarkdown規則に則ったプレーンテキストファイルとして保存されます。
2. **ハイブリッドストレージ**:
   * **デモモード**: メモリ上で手軽に動作検証が可能です。
   * **ローカルフォルダ連携**: ブラウザの File System Access API を用い、ローカルPC内のファイルを直接読み書きします。
   * **サーバー連携モード**: セキュリティで保護された 独自API を経由し、データをサーバーの公開領域外に安全に隔離して同期・管理します（インメモリキャッシュモデルによる超高速同期）。
3. **キーボード ＆ モバイルフレンドリー**: 充実したキーボードショートカットに加え、スマホでの左右スワイプ切り替えや Pull-to-refresh ジェスチャーをサポートします。

## 3. Markdownデータ構造（規定フォーマット）

システムが正しくエントリー（Markdownファイル）を解釈・編集・シリアライズするために、以下の厳格なフォーマットを規定しています。この形式から逸脱したMarkdownファイルは **「編集不可（要手動修正）」** として扱われ、読み込み時にエラーバナーが表示されます。

### エントリーのMarkdownテンプレート

```markdown
# [エントリータイトル]

## 概要（または指定した任意の項目名）

[ここに概要の文章が入ります。改行やマークダウン箇条書きも可能]

## ブロック

### [ブロックタイトル (任意)] <!-- [一意のブロックID(半角英数字5文字)] [タイムスタンプ (YYYY-MM-DD HH:mm:ss)] -->
[ここにブロックの本文が入ります。Markdown形式で複数行記述可能]

### [別のブロックタイトル (任意)] <!-- [別のブロックID] [タイムスタンプ] -->
[ブロック本文]
```

### パース ＆ バリデーションルール
1. **H1タイトル**: ファイルの先頭は必ず `# タイトル名` で始まらなければなりません。
2. **概要セクション**: 2番目のセクション見出しは必ず `## 概要` (またはカスタマイズされた `## タイトル`) であり、その下に本文が続きます。
3. **ブロックセクション**: 3番目のセクション見出しは必ず `## ブロック` でなければなりません。これ以降の記述はすべて個別ブロックカードとして扱われます。
4. **個別ブロック (H3)**:
   * 見出しレベルは必ず `###` (H3) です。
   * 見出しの末尾には、必ず HTML コメント形式のメタデータ `<!-- [blockId] [YYYY-MM-DD HH:mm:ss] -->` が付与されている必要があります。
   * ブロックIDは半角英数字5文字（例: `a73bc`）で重複しない必要があります。タイムスタンプは `YYYY-MM-DD HH:mm:ss` 形式です。

## 4. ストレージ層設計

`storage.js` で提供されるストレージクラスは、ポリモーフィズムに基づいて同一のインターフェースを持ち、アプリ起動時またはフォルダ選択時に `currentStorage` に差し替えられます。

### ストレージクラス共通インターフェース

| メソッド / プロパティ | 型 | 説明 |
| :--- | :--- | :--- |
| `favorites` | `Set<string>` | お気に入りキー（`category/file.md`）を保持するSetオブジェクト |
| `init()` | `Promise<boolean>` | ストレージの初期化。`ServerStorage` の場合はサーバーから全データを一括取得 |
| `listCategories()` | `Promise<string[]>` | カテゴリー（フォルダ名）の一覧をソートして返却。「ゴミ箱」は自動で末尾にソート |
| `listEntries(cat)` | `Promise<object[]>` | 指定カテゴリー内のエントリーファイル名・更新日時（`mtime`）の一覧を返却 |
| `readEntry(cat, file)` | `Promise<string>` | Markdownファイルの中身をプレーンテキストで返却 |
| `writeEntry(cat, file, md)` | `Promise<boolean>` | 既存のエントリーに上書き書き込み |
| `createEntry(cat, file, md)`| `Promise<string>` | 指定カテゴリーにファイルを新規作成。重複時は自動で連番（`-2.md`等）を解決し、**確定したファイル名文字列**を返却 |
| `moveEntry(oc, of, nc, nf)` | `Promise<string>` | ファイルを別カテゴリーや別名に移動・リネーム。重複解決を行い、**移動後の確定ファイル名文字列**を返却 |
| `deleteEntry(cat, file)` | `Promise<boolean>` | ファイルを物理削除 |
| `isFavorite(cat, file)` | `boolean` | 指定ファイルがお気に入りに入っているか判定 |
| `toggleFavorite(cat, file)`| `Promise<boolean>` | お気に入り状態を反転させて同期保存 |
| `saveFavorites()` | `Promise<void>` | お気に入り情報を永続化 |

## 5. サーバーサイドAPIの概要設計

サーバー連携モードで動作する場合、フロントエンドは `ServerStorage` から各種APIリクエストを送信します。APIはホスティングサーバー上に設置され、以下の仕様を満たしている必要があります。

### APIの基本仕様・安全設計
1. **ステートレス / JSON通信**: APIはステートレスであり、入力・出力ともにJSON形式（POST時）で通信を行います。
2. **トラバーサル防御**: カテゴリー名やファイル名に対するトラバーサル攻撃 (`..`, `/`, `\\`, `\0`) の排除を必須とします。
3. **アトミック書き込み**: 複数端末からのアクセスによるファイル破損を防ぐため、一時ファイルを書き出してから名前を変更する（アトミック置換）アルゴリズムを導入します。
4. **競合検出（楽観的ロック）**: ハッシュ値（revision）を用いた上書き衝突防止策を導入します。

※具体的な通信スキーマ、および詳細な実装手引きについては、ルートディレクトリの **`specification_for_api.md`** を参照してください。

## 6. フロントエンド UI/UX ＆ ナビゲーション仕様

### パンくずリストナビゲーション
詳細ビューのタイトルヘッダーは3階層（`すべて` ＞ `[カテゴリー名]` ＞ `[タイトル]`）で描画されます。
* 「すべて」をクリック ──> すべてのエントリーリストを表示。
* 「カテゴリー名」をクリック ──> そのカテゴリーのエントリーリストを表示。

### お気に入り（Pickup）とゴミ箱ガード
* **ゴミ箱ガード**: エントリーが「ゴミ箱」または「trash」カテゴリーに配置されている場合：
  * 詳細ビューのお気に入りトグルボタンは 🗑️ アイコンに変わり、無効化（disabled）されます。
  * ショートカットキー（Sキー）やお気に入りトグルAPIの呼び出しはガード処理によって強制早期リターンされます。
  * リスト画面上でも星マーク（★/☆）の代わりに 🗑️ が表示され、クリックできません。

### キャッシュと手動リロード（同期）
* サーバーモード起動時は `ServerStorage` が全データをメモリ上に保持して高速化するため、外部による更新を取り込むための「手動リロード機能」を用意しています。
* PC環境では、ヘッダーに配置されたリロードボタン（🔄）をクリックすると、ボタンが回転アニメーションをしながらサーバーのデータを再同期します。
* モバイル環境では、スクロールが最上部（`scrollTop === 0`）の状態で画面を下方向に引っ張ることで **Pull-to-refresh インジケーター** が表示され、規定しきい値（75px）以上引っ張って指を離すと、インジケーターが回転しながら同期を実行します。

### タッチスワイプジェスチャー
スマホ等での片手操作性を向上するため、メインコンテンツエリア（左右方向へのスワイプ量 80px 以上）で「リストビュー」と「最後に開いていたエントリー詳細ビュー」を交互に切り替えるトグルスワイプジェスチャーをサポートします。入力欄がアクティブな場合は、誤動作防止のためスワイプは無視されます。

### キーマップのカスタマイズ
ユーザーはキー割り当てを独自にカスタマイズできます。
* `app/src/keymaps.js.sample` をコピーして `app/src/keymaps.js` を作成します（`keymaps.js` は `.gitignore` により Git 管理外です）。
* `window.EntryMemo.Keymaps` オブジェクトに対象のアクションキー割り当てを記述することで、デフォルト設定を任意に上書きできます。
* `keymaps.js` が存在しない場合は、標準のデフォルトキーボードショートカットで透過的にフォールバック動作します。

## 8. 安全性・排他制御（アトミック置換・競合解決）

データの破損や消失を防ぐため、以下の安全機構が実装されています。

### アトミック置換
サーバーへの書き込み時、API側は直接既存ファイルを開いて上書きするのではなく、まず一時ファイルを書き出します。書き込み成功を確認した後にアトミックに対象ファイルを置換します。これにより、書き込み途中のネットワーク切断やサーバーダウン時でも、既存データが破損するのを防ぎます。

### 楽観的ロック（競合検出）
複数の端末から同一のエントリーを同時に編集したことによる上書き消失（Lost Update）を防ぐため、ハッシュ値（revision）を用いた競合検出を行います。
1. エントリー読み込み時、サーバーは現在のMarkdownファイルの中身の SHA-256 ハッシュ値を `revision` として返却し、クライアントはこれを保持します。
2. 保存時、クライアントは保持していた `baseRevision` をリクエストに含めて送信します。
3. サーバーは現在のファイルハッシュを再度計算し、`baseRevision` と不一致である場合は、第三者によって先にファイルが更新されたと判定し書き込みを拒否します。
